# 初期要件メモ: Java grep結果 自動分類ツール

## やりたいこと

Javaプロジェクトに対してある特定の文言（文字列リテラル・定数値など）をgrepすると、ヒット数が4万行規模になることがある。
それらを1件ずつ目視確認するのは非現実的なため、各ヒット行を「Javaにおける使用パターン」に自動分類して一覧化したい。

### 背景・目的

文言を別の用途に転用・削除したい。そのため、現在どこでどのように使われているかを**全件もれなく洗い出す**必要がある。

---

## 分析の2段階構造（重要）

grep結果だけでは「文言がどう使われているか」を完全には把握できない。
以下の2段階で追跡する必要がある。

### 第1段階: 直接参照（grep結果から）

文言のリテラルが直接コードに現れているもの。grepで拾える。

```java
// ← grep でここがヒットする
public static final String CODE = "TARGET_VALUE";
String msg = "TARGET_VALUE";
if (x.equals("TARGET_VALUE")) { ... }
```

### 第2段階: 間接参照（変数・定数経由）

第1段階で「変数代入」または「定数定義」と分類されたものについて、
**その変数/定数がその後どこでどのように使われているか**を追跡する。
grepでは拾えないが、実際の「使われ方」はこちらにある。

```java
// 第1段階: grep でここがヒット（定数定義）
public static final String CODE = "TARGET_VALUE";   // ← Constants.java:10

// 第2段階: CODEがここで使われている（条件判定）← grep では出てこない
if (someVar.equals(CODE)) { ... }                   // ← Service.java:110

// 第2段階: CODEがここで使われている（メソッド引数）← grep では出てこない
someService.process(CODE);                           // ← Handler.java:55
```

**→ この両方をTSVに出力し、どちらの参照か区別できるようにする。**

---

## 入力

### grep実行方法（ユーザーが手動で実行）

```bash
grep -rn "検索文言" /path/to/javaproject > input/検索文言.grep
```

- ファイル名（拡張子なし）= 検索した文言名
- 複数の文言を調べたい場合は、ファイルを複数作成して `input/` に配置する
- grep行フォーマット: `ファイルパス:行番号:コード行`
  - 例: `src/main/java/Foo.java:42:    String msg = "ERROR";`

### ソースコードの配置

- grep結果ファイルと同じサーバ上の別ディレクトリに、grep対象のJavaソースコードが存在する
- ツール実行時にソースコードのルートディレクトリをパスで指定する
- 第2段階の間接参照追跡に使用する

### ツールの起動方法

```bash
run.sh --source-dir /path/to/javaproject
```

- `--source-dir`: Javaソースコードのルートディレクトリ（必須）
- `--input-dir`: grep結果ファイルの配置ディレクトリ（省略時: `input/`）
- `--output-dir`: TSV出力先ディレクトリ（省略時: `output/`）

### Javaソースファイルの文字コード

- 対象Javaプロジェクトの文字コードは **Shift-JIS**
- ソースファイル読み込み時は `encoding='shift_jis'` を使用する
- エンコーディングエラーが発生した場合は `errors='replace'` で継続し、レポートに記録する

### 入力ファイル配置

```
input/
├── ERROR_CODE.grep    # "ERROR_CODE" を grep した結果
└── STATUS_OK.grep     # "STATUS_OK" を grep した結果
```

## 出力

### フォーマット: TSV（タブ区切り）

- エンコード: UTF-8 BOM付き（Excelで文字化けしないよう）
- 1行1件（直接参照 + 間接参照の全件）
- **ソート順**: 文言 → ファイルパス → 行番号（昇順）

### 列定義

| 列名 | 内容 |
|---|---|
| 文言 | 検索した文言（入力ファイル名から取得） |
| 参照種別 | `直接` または `間接` |
| 使用タイプ | 自動分類した使用パターン（下記7種） |
| ファイルパス | 該当行のファイルパス |
| 行番号 | 該当行の行番号 |
| コード行 | 該当行のコード（前後の空白はtrim） |
| 参照元変数名 | 間接参照の場合、経由した変数/定数名（直接参照は空欄） |
| 参照元ファイル | 間接参照の場合、変数/定数が定義されたファイルパス（直接参照は空欄） |
| 参照元行番号 | 間接参照の場合、変数/定数が定義された行番号（直接参照は空欄） |

### 出力例

```
文言	参照種別	使用タイプ	ファイルパス	行番号	コード行	参照元変数名	参照元ファイル	参照元行番号
TARGET	直接	定数定義	Constants.java	10	public static final String CODE = "TARGET"
TARGET	間接	条件判定	Service.java	110	if (someVar.equals(CODE)) {	CODE	Constants.java	10
TARGET	間接	メソッド引数	Handler.java	55	someService.process(CODE);	CODE	Constants.java	10
TARGET	直接	メソッド引数	Util.java	30	log("TARGET");
TARGET	直接	条件判定	Validator.java	80	if (x.equals("TARGET")) {
```

### 出力ファイル配置

```
output/
└── TARGET.tsv   # input/TARGET.grep に対応するTSV
```

## 自動分類の種類（7種）

直接・間接ともに同じ分類を適用する。

| 使用タイプ | 内容 |
|---|---|
| アノテーション | `@Annotation("値")` または `@Annotation(変数)` |
| 定数定義 | `static final TYPE NAME = "値"` |
| 変数代入 | ローカル変数・フィールドへの代入 |
| 条件判定 | `if/else if/while` の条件式、`.equals()`, `==`/`!=` 比較 |
| return文 | `return "値"` または `return 変数` |
| メソッド引数 | メソッド呼び出しの引数として渡している |
| その他 | 上記に当てはまらないもの（コメント行・文字列連結など）。コメント内のヒットも除外せず含める |

## 分析方針

### 第1段階: 直接参照の分類

- grep結果のファイルパス＋行番号でソースファイルを開く
- `javalang` でAST解析し、使用タイプを分類
- 参照種別 = `直接`、参照元列は空欄

### 第2段階: 間接参照の追跡

第1段階で「定数定義」または「変数代入」に分類された行に対して実施：

1. ASTから変数名/定数名を抽出（例: `CODE`）
2. 変数の種類に応じて**追跡スコープを変える**（下記参照）
3. 各出現箇所をAST解析して使用タイプを分類
4. 参照種別 = `間接`、参照元列に元の定義情報を記録
5. ただし「定義行自体」は第1段階で既に記録済みなので重複して追加しない

#### 追跡スコープの判断基準

grepキーワードは `"777"` のような**文字列の値そのもの**であることが多い。
その値が代入された変数の種類によって、同名変数の誤ヒットリスクが異なるため、スコープを分けて追跡する。

| 変数の種類 | 例 | 追跡スコープ | 理由 |
|---|---|---|---|
| 定数（`static final`） | `public static final String CODE = "777"` | **プロジェクト全体** | 名前が意味を持ち誤ヒットが少ない |
| インスタンス変数（フィールド） | `private String type = "777"` | **同一クラス内** + **getter経由でプロジェクト全体** | クラス外からはgetterで参照される |
| ローカル変数 | `String s = "777"` | **同一メソッド内** | `s` や `type` など汎用名が多く全体検索は誤ヒットが大量発生する |

#### フィールドのgetter追跡（第3段階）

フィールドに代入された値はgetter経由でクラス外に渡ることがある。
**もれなく洗い出す**ため、以下の手順でgetter呼び出し箇所もプロジェクト全体追跡する。

**ステップ1: 同一クラス内のgetter候補を特定（AST解析）**

以下2つの方法を併用して、非標準命名のgetterも拾う：
- **命名規則**: フィールド名 `type` → `getType()` を探す
- **return文解析**: `return type;` しているメソッドを全て拾う（`fetchOrderType()` 等の非標準名も対象）

**ステップ2: プロジェクト全体でgetter呼び出しを検索・分類**

- `.getType(` のようなパターンでプロジェクト全体を検索
- 各呼び出し箇所をAST解析して使用タイプを分類
- TSVに `参照種別 = 間接（getter経由）` として出力

**false positiveについて**

`getType()` は他のクラスにも存在しうるため、無関係なクラスのgetter呼び出しが混入する可能性がある。
ただし「もれなく」が最優先のため、**疑わしいものは全て出力する**方針とする。
人間がTSVの `参照種別` 列でフィルタして確認できる。

**setterは追跡不要**

`obj.setType(someVar)` の `someVar` に "777" が渡る場合は、`someVar` 自体がすでに直接参照または変数追跡で拾えているため、setter経由の逆追跡は不要。

### 使用ライブラリ

- **javalang**: Java 7以上のソースファイルをAST解析
- **フォールバック**: javalangがパースエラーを起こした場合は正規表現で分類

## 配布・実行環境

- **言語**: Python 3.8以上
- **依存ライブラリ**: `javalang`（外部パッケージ1つのみ）
- **配布形式**: zip
  - 解凍 → `setup.sh`（venv作成＋javalangインストール）→ `run.sh`（実行）で即使用可能
  - Windows用: `setup.bat` / `run.bat` も提供
- **動作OS**: Windows / Mac / Linux（クロスプラットフォーム）

## 非機能要件

### 最優先: もれなく洗い出すこと

**処理時間より網羅性を優先する。** 時間がかかっても全件正確に分類できることが最重要。

- 直接参照・間接参照ともに1件も見落としてはならない
- 分類できないものは「その他」に確実に落とす（スキップしない）
- 処理時間の目標は設けない（数分かかっても許容）

### エラー耐性

- 不正なgrep行（バイナリ通知・空行など）は自動スキップし、**スキップした件数・内容をレポート**
- エンコーディングエラーが発生しても処理継続（置換文字で対応）
- javalang のパースエラーが発生しても処理継続（正規表現フォールバック）
- フォールバックが発生したファイル名・行番号もレポートに記録する

### パフォーマンス（補助的）

- ASTキャッシュで同一ファイルの再解析を省略（処理時間短縮のための工夫は行う）
- ただしキャッシュ処理が網羅性を損なう場合は採用しない
