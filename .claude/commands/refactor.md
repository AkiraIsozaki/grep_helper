# Javaリファクタリング (完全自動実行モード)

**重要:** このワークフローは、ユーザーの介入なしに、開始から完了まで完全に自動で実行されるように設計されています。各ステップは完了後、ただちに次のステップへ移行してください。思考の途中でユーザーに確認を求めたり、作業を中断したりしないでください。

**引数:** リファクタリング対象と目的 (例: `/refactor StepViewModelのプリミティブ執着を解消`)

---

## ステップ1: 準備とコンテキスト設定

1. 現在のタスクコンテキストを確立する:
  - 対象: `[引数で与えられたリファクタリング対象]`
  - 日付: `[現在の日付をYYYYMMDD形式で取得]`
  - ステアリングディレクトリパス: `.steering/[日付]-refactor-[対象キーワード]/`
2. 上記ステアリングディレクトリを作成する。
3. 以下の3つの空ファイルを作成する:
  - `[ステアリングディレクトリパス]/requirements.md`
  - `[ステアリングディレクトリパス]/design.md`
  - `[ステアリングディレクトリパス]/tasklist.md`

## ステップ2: プロジェクト理解と既存ドキュメントの読み込み

1. `CLAUDE.md`を読み、プロジェクトの全体像を把握する。
2. `Skill('refactor-java')`をロードして、リファクタリングの手順・観点・パターンを把握する。
3. 以下の永続ドキュメントを読み、プロジェクトのアーキテクチャと規則を理解する:
  - `docs/architecture.md`
  - `docs/development-guidelines.md`
  - `docs/functional-design.md`（対象機能に関係する場合）

## ステップ3: 対象コードの分析

1. Grepツールを使用し、対象クラス・メソッドに関連するキーワードでソースコード(`src/`)を検索する:
  ```
  Grep('[対象に関連するキーワード]', 'src/')
  ```
2. 対象コードを読み込み、`Skill('refactor-java')`の9つの観点で問題を洗い出す:
  - 観点1: 関心の分離とパッケージ可視性（publicを外すと壊れるものだけが本当の公開API）
  - 観点2: 完全コンストラクタと不変性（setter、mutable フィールドの有無）
  - 観点3: プリミティブ執着（int/Stringで表現されたドメイン概念）
  - 観点4: 名前設計（略語・汎用名・booleanメソッドの命名）
  - 観点5: インターフェースの利用（具象クラスへの直接依存）
  - 観点6: ドキュメントとの乖離（docs/の設計と実装の不一致）
  - 観点7: 重複コードの抽出（同じ構造のtry-catch・nullチェックが3箇所以上）
  - 観点8: メソッドの長さと複雑度（20行超 / ネスト2段超 / 複数の責務）
  - 観点9: 例外処理の設計（catch(Exception)の広すぎる補足・null返却・不整合なポリシー）
3. 分析結果をステアリングファイルに記録する:
  - `requirements.md`: 現状の問題点と変更目標（変更なしの場合はその理由）
  - `design.md`: 変更アプローチと実装方針
  - `tasklist.md`: ソース修正タスクと**対応するドキュメント更新タスクを1対1で対応付ける**

**分析の結果、変更が不要と判断した場合:** tasklist.mdに「変更対象なし」と理由を記録し、ただちに**ステップ5**へ進む。これはリファクタリングの失敗ではなく、現在の設計が適切であるという確認です。

## ステップ4: 実装ループ (tasklist.mdの完全消化)

**このステップは、`tasklist.md`の全タスクが `[x]` になるまで自動で繰り返されるループ処理です。**
**このステップが正常に完了したら、決して停止せず、ただちにステップ5に進むこと。**

各タスクは **1タスク = コード変更 + テスト確認 + ドキュメント更新** のサイクルで完了させる。ドキュメント更新は後回しにせず、コード変更直後に行う。

**ループ開始:**

1. タスクリストの読み込み:
  - `[ステアリングディレクトリパス]/tasklist.md`ファイルを読み込む。

2. 進捗の確認:
  - ファイル内に未完了タスク (`[ ]`) が存在するか確認する。
  - **もし未完了タスクが存在しない場合:** この実装ループは完了とみなし、ただちに**ステップ5**へ進む。
  - **もし未完了タスクが存在する場合:** 次の処理（3. タスクの実行）に進む。

3. タスクの実行（1タスク完了サイクル）:
  - `tasklist.md`の**先頭にある未完了タスク**を1つ特定する。
  - コードを変更する（`Skill('refactor-java')`の実装パターンに従う）。
  - `./gradlew test`でグリーンを確認する（失敗したら即修正してからドキュメント更新へ進む）。
  - **このタスクに対応する`docs/`を更新する**（絶対に後回しにしない）。
    - 新しい型・Value Object → `functional-design.md`と`glossary.md`
    - パッケージ構成の変更 → `architecture.md`と`repository-structure.md`
    - 命名の変更 → `development-guidelines.md`と`glossary.md`
    - 可視性ルールの変更 → `development-guidelines.md`
    - 重複コードをヘルパーに抽出 → `development-guidelines.md`（実装原則）
    - メソッド分割・クラス分割 → `architecture.md`（責務変更）
    - 例外処理方針の統一 → `development-guidelines.md`（エラーハンドリング方針）

4. タスクリストの更新:
  - `Edit`ツールを使用して`tasklist.md`の該当タスクを `[ ]` から `[x]` に変更する。

5. ループ継続:
  - **ステップ4の先頭 (1. タスクリストの読み込み) に戻り、処理を繰り返す。**

---
### ※ 実装ループ内の例外処理ルール ※

実装ループの実行中に以下の状況が発生した場合は、このルールに従って自律的に対処し、ループを継続すること。

- **ルールA: タスクが大きすぎる場合**
  - **対処法:** 現在のタスクをより小さな複数のサブタスクに分割する。`Edit`ツールを使い、元のタスクを削除し、その場所に新しいサブタスク（`[ ]`付き）を挿入する。その後、ループを継続する。

- **ルールB: 技術的理由でタスクが不要になった場合**
  - **条件:** 実装方針の変更、アーキテクチャの変更、依存関係の変更など、明確な技術的理由がある場合にのみ適用。
  - **対処法:** `Edit`ツールを使い、該当タスクを `[x] ~~タスク名~~ (理由: [具体的な技術的理由を簡潔に記述])` の形式で更新する。その後、ループを継続する。

- **ルールC: パッケージ可視性変更時の注意**
  - `public`を外す前に、必ずGrepでパッケージ外からの使用箇所を確認する。
  - 使用箇所が存在する場合は変更しない（それが本当の公開APIである）。

- **❌ 絶対禁止の行為:**
  - 未完了タスクを「後でやる」「別タスクにする」などの理由で意図的にスキップすること。
  - ドキュメント更新を後回しにしてコード変更だけ進めること。
  - 機能を変更すること（リファクタリングは動作を変えない）。
  - ユーザーに判断を仰ぐこと。

---

## ステップ5: 最終検証

1. 全タスクが完了していることを確認する（`tasklist.md`の全行が`[x]`）。
2. 以下のコマンドを順番に実行し、全てがパスすることを確認する:
  ```bash
  Bash('./gradlew test')    # 全テストグリーン
  Bash('./gradlew check')   # Checkstyle 0件
  Bash('./gradlew build')   # ビルド成功
  ```
3. いずれかのコマンドでエラーが発生した場合は、問題を分析し、修正コードを生成・適用してから、再度このステップを実行する。

**このステップが正常に完了したら、決して停止せず、ただちにステップ6に進むこと。**

## ステップ6: 振り返りとドキュメント最終確認

1. `Skill('steering')`を**振り返りモード**で実行し、`tasklist.md`に申し送り事項を記載する:
  - 実施した変更の概要（「変更なし」の場合はその理由）
  - 計画と実績の差分（計画になかった変更や、やめた変更）
  - 次回のリファクタリングへの示唆

2. `tasklist.md`の全タスクが`[x]`になっているか、古くなった記述が`docs/`に残っていないかを最終確認する。

3. `glossary.md`に追加した新しい概念の用語が登録されているかを確認する。

## 完了条件

このワークフローは、以下の全ての条件を満たした時点で自動的に完了となる。

- ステップ4: `tasklist.md`の全てのタスクが完了状態（`[x]`または技術的理由での正当なスキップ）になっている。
- ステップ5: `test`, `check`, `build`の全てのGradleタスクがエラーなく成功する。
- ステップ6: `tasklist.md`に振り返りが記載されている。

この完了条件を満たすまで、自律的に思考し、問題解決を行い、作業を継続すること。
