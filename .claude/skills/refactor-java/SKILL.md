# Javaリファクタリングスキル

このスキルは、**ソースコードとドキュメントの完全な整合性**を保ちながら、
保守性・拡張性・設計品質を向上させるJavaリファクタリングを実施します。

表面的なコード修正で終わらせず、設計の意図をドキュメントに反映させることが核心です。
**1つのタスクを完了したら、対応するドキュメント更新もその場で行います**（後回しにしない）。

---

## フェーズ0: 準備（必ず最初に実施）

### 既存ドキュメントを読む

```
Read('docs/architecture.md')
Read('docs/development-guidelines.md')
Read('docs/functional-design.md')  # 機能設計が変わる場合
```

これらはプロジェクトの「北極星」です。リファクタリングはこのドキュメントが定める
アーキテクチャ・命名規則・実装原則の範囲内で行います。

### ステアリングファイルを作成

`Skill('steering')` を使い、`.steering/YYYYMMDD-refactor-[対象]/` に3ファイルを作成します。

tasklist.mdには以下を含めること：
- 分析結果（現状の問題点リスト、または「変更不要」の結論）
- ソース修正タスク（ファイル単位）— 変更がない場合はその旨を記録
- 各タスクに対応するドキュメント更新（タスクと1対1で対応付ける）
- 検証タスク（テスト・checkstyle・build）

---

## フェーズ1: 分析（何が問題かを明確にする）

対象コードを読み、以下の9つの観点で問題を洗い出します。
詳細なパターンと判断基準は `references/java-patterns.md` を参照してください。

**分析の結果、変更が不要な場合も正当な結論です。** その場合は理由を tasklist.md に記録し、
フェーズ2の実装をスキップしてフェーズ4（検証）に進みます。

### 観点1: 関心の分離とパッケージ可視性

**問いかけ**: 「このクラスは本当に外から見える必要があるか？」

- 同じパッケージ内でのみ使われる型・メソッドが `public` になっていないか
- **必ず Grep でパッケージ外からの使用箇所を確認してから変更する**
- パッケージをまたぐ依存が多すぎないか
- 凝集すべきクラス群が別々のパッケージに散っていないか

**判断基準**: `public` を外すと壊れるものだけが本当の公開APIです。

### 観点2: 完全コンストラクタと不変性

**問いかけ**: 「このオブジェクトは生成後に状態が変わるか？変わる必要があるか？」

- setter / `setXxx()` メソッドが存在する → 完全コンストラクタへの移行候補
- フィールドに `final` がついていない → 不意の再代入リスク
- 不要な `static` メソッド・フィールドが増えている

**なぜ重要か**: 不変オブジェクトはスレッドセーフで、テストが簡単で、バグが起きにくい。

### 観点3: プリミティブ執着

**問いかけ**: 「このint/Stringは、ビジネス上の意味を持つ概念か？」

- `int userId` → `UserId userId` にできないか
- `String status` → `Status` enumにできないか
- `long[]` で座標・範囲を表現している → Value Objectへ
- メソッドに `int row, int col` が何度も現れる → 専用型に包む

**Value Object の型選択**:
- **Java 16+ の `record` を優先する** — `equals`/`hashCode`/`toString` が自動生成され簡潔
- 自己検証ロジック（コンストラクタでの範囲チェック等）が必要な場合は `final class`

**⚠️ テスト内部クラス名の衝突**: Value Object 導入時、テストファイル内に同名の内部クラスが存在すると
コンパイルエラーになります。分析フェーズで Grep してクラス名の衝突を確認してください。

### 観点4: 名前設計

**問いかけ**: 「この名前だけでビジネスロジックが伝わるか？」

- 略語・汎用名（`data`, `info`, `tmp`, `helper`, `util`, `manager`）
- boolean を返すメソッドが `is-/has-/can-/should-` で始まっていない
- メソッド名が処理の「what」ではなく「how」を表している

### 観点5: インターフェースの利用

**問いかけ**: 「この依存は具象クラスに対してか、抽象（インターフェース）に対してか？」

- テスト時にモックしたい処理が具象クラスに直接依存している
- **過度な抽象化も避ける**。実装が2つ以上存在するか、テスタビリティのために必要な場合のみ導入

### 観点6: ドキュメントとの乖離

**問いかけ**: 「このクラスの責務はdocs/に書かれた設計と一致しているか？」

- `docs/architecture.md` のレイヤー依存ルールに違反していないか
- 追加された機能がドキュメントに反映されていないか

### 観点7: 重複コードの抽出

**問いかけ**: 「このコードブロック、別の場所でも見たことがあるか？」

- 同じ構造の try-catch + エラーダイアログが複数メソッドに出現 → ヘルパーメソッドへの抽出候補
- null チェックが同じパターンで繰り返される → 共通処理に抽出、または `Optional` の活用
- 似たような条件分岐の連鎖が複数箇所に散らばっている → テンプレートメソッドパターン

**判断基準**: 「3回以上コピーが存在するパターンは名前を付けて抽出する」(Rule of Three)。
**やりすぎに注意**: 一度しか使われない処理を無理に抽出しない。

### 観点8: メソッドの長さと複雑度

**問いかけ**: 「このメソッドは1つのことをしているか？テスト時に1行で説明できるか？」

- **20行を超えるメソッド** → 意図を表す名前の private メソッドへの抽出を検討
- **ネストが2段以上の if/for** → 早期 return または extracted method で平坦化
- **メソッド名に `and` / `or` / `then` が入る** → 複数の責務を担っているサイン
- **`EigenvalueCalculator.java` が400行超** → クラス分割の検討候補

**目安**: メソッド本体 20〜30 行以内、クラス 300 行以内（`docs/development-guidelines.md` のファイルサイズ方針と整合）。

### 観点9: 例外処理の設計

**問いかけ**: 「この例外処理は必要最小限の粒度か？同じパターンが一貫しているか？」

- **`catch (Exception e)` や `catch (Throwable e)`** → 本当に必要な例外型に絞る
  （過広な catch は `NullPointerException` や `StackOverflowError` まで飲み込む）
- **同類のメソッドで catch する例外型が異なる** → 一貫性のある例外ポリシーに統一
- **null を返している** → `Optional<T>` への置き換えを検討
  （`null` は「値なし」の意図が伝わらず、呼び出し側の null チェック漏れを招く）
- **例外のメッセージが空または技術的すぎる** → 原因と対処法を含む日本語メッセージへ

**対象外**: JavaFX の `@FXML` イベントハンドラーは checked exception を直接 throw できないため、
catch-and-display パターン自体は正当。問題は catch する例外型の粒度。

## フェーズ2: 実装とドキュメント更新（1タスク = コード + ドキュメント）

### 実装の順序（依存関係を考慮）

1. **コアレイヤー（core/）から始める** — 上位レイヤーが依存するため先に安定させる
2. **インターフェース定義** — 実装前に契約（interface）を確定させる
3. **Value Object の導入** — プリミティブを包む新型の追加（既存型の変更前に）
4. **可視性の絞り込み** — `public` → パッケージプライベートへの変更
5. **上位レイヤー（service → viewmodel → view）の順に修正**

### 各タスクの完了サイクル（これを繰り返す）

```
① コードを変更する
② ./gradlew test でグリーンを確認（失敗したら即修正）
③ このタスクに対応する docs/ を更新する（後回しにしない）
④ tasklist.md の該当タスクを [x] に更新する
⑤ 次のタスクへ
```

**ドキュメントをその場で更新する理由**: 実装の最後にまとめて更新しようとすると、
コンテキストが足りなくなった場合にドキュメント更新がスキップされます。
1タスクが終わるたびに対応するドキュメントも完結させることで、
部分的な状態でも「実装とドキュメントが常に一致している」を保証します。

### タスクに対応するドキュメント更新の早見表

| 変更の種類 | 更新する docs/ |
|-----------|----------------|
| 新しい型・Value Object の追加 | `functional-design.md`（型の記述）、`glossary.md`（用語追加） |
| パッケージ構成の変更 | `architecture.md`（レイヤー構成図）、`repository-structure.md` |
| 命名の変更（クラス名・メソッド名） | `development-guidelines.md`（命名規則例）、`glossary.md` |
| 可視性ルールの変更 | `development-guidelines.md`（実装原則） |
| 新しい設計パターンの採用 | `architecture.md`（アーキテクチャパターン） |
| 重複コードを共通ヘルパーに抽出 | `development-guidelines.md`（実装原則にヘルパーのルール追記） |
| メソッド分割・クラス分割 | `architecture.md`（対象クラスの責務変更）、`repository-structure.md`（新ファイル追加） |
| 例外処理方針の統一 | `development-guidelines.md`（エラーハンドリング方針） |

### 実装パターン（コードスニペット）

**パッケージプライベート化**:
```java
// Before: publicだが実際にはパッケージ内でのみ使用
public class InternalHelper { ... }

// After: アクセス修飾子を削除（同じパッケージからのみアクセス可能）
class InternalHelper { ... }
```

**完全コンストラクタへの移行**:
```java
// Before: setter経由で状態を設定
Result result = new Result();
result.setValue(42);

// After: コンストラクタで全フィールドを初期化
Result result = new Result(42, "...");
```

**Value Object（record を優先）**:
```java
// ✅ Java 16+ record — equals/hashCode/toString が自動生成される
record CellPosition(int row, int col) {
    CellPosition {
        if (row < 0 || col < 0) throw new IllegalArgumentException("...");
    }
}

// final class が必要なのは: パッケージプライベートにしたい、
// または record では表現できない複雑な検証がある場合
final class CellPosition {
    private final int row;
    private final int col;
    CellPosition(int row, int col) { ... }
    int row() { return row; }
    int col() { return col; }
}
```

詳細なパターン・アンチパターンは `references/java-patterns.md` を参照してください。

## フェーズ3: 最終整合確認

全タスクの実装とドキュメント更新が終わったら、全体を俯瞰して確認します。

- tasklist.md の全タスクが `[x]` になっているか
- 変更に対応するドキュメントが全て更新されているか（早見表と照合）
- 古くなった記述（旧クラス名・旧型・旧アーキテクチャ）が残っていないか
- `glossary.md` に追加した新しい概念の用語が登録されているか

## フェーズ4: 検証

```bash
./gradlew test         # 全テストグリーン
./gradlew check        # Checkstyle 0件
./gradlew build        # ビルド成功
```

いずれかが失敗した場合は、フェーズ2に戻って修正します。

## フェーズ5: 振り返り

`Skill('steering')` の振り返りモードを使い、tasklist.mdに記録：

- 実施した変更の概要（「変更なし」の場合はその理由）
- 計画と実績の差分（計画になかった変更や、やめた変更）
- 次回のリファクタリングへの示唆

## 重要な判断基準

### 「変更なし」も正当な完了

分析の結果、変更が不要だった場合は：
- tasklist.md に「変更対象なし」と理由を記録する
- フェーズ4（検証）でビルドが通ることを確認する
- フェーズ5（振り返り）で分析の結果を記録する

これはリファクタリングの失敗ではなく、**現在の設計が適切であるという確認**です。

### やりすぎないために

- 依頼範囲外のクラスの変更（スコープを守る）
- 将来の要件を先取りした抽象化（YAGNI: You Aren't Gonna Need It）
- 機能を変えること（リファクタリングは動作を変えない）
- すべての `public` を消すこと（公開APIは必要）

### 迷ったときの問いかけ

「この変更で、**次にここを修正する人**の仕事が楽になるか？」

答えがYesなら変える。Noなら変えない。
